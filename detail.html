💬 3. Gemini Prompt แบบเต็ม

GEMINI_PROMPT เป็น f-string ที่ประกอบด้วยชื่อ Type จาก LOAN_TYPES โดยอัตโนมัติ (ถ้าเปลี่ยนชื่อ Type ใน config, Prompt จะอัปเดตตามทันทีโดยไม่ต้องแก้ข้อความ)

main.py:79-119
GEMINI_PROMPT = f"""You are a document data extraction and verification system for a loan department.
You will receive a scanned document (PDF or photo/image, possibly in Thai or English).
It is a Thai student-loan confirmation form (แบบยืนยันการเบิกเงินกู้ยืม กองทุนเงินให้กู้ยืมเพื่อการศึกษา).

Read the document carefully and extract EXACTLY these fields:

1. "name": ชื่อ-นามสกุลผู้กู้ (หลังคำว่า "ข้าพเจ้า")
2. "id_card_top": เลขบัตร 13 หลัก บริเวณบาร์โค้ดด้านบน (DIGITS ONLY)
3. "id_card_body": เลขบัตร 13 หลัก ที่บรรทัด "เลขบัตรประจำตัวประชาชน" ในส่วนที่ 1
4. "loan_type": อ่านจากบรรทัดที่ขึ้นต้นด้วย "ลักษณะที่" แล้วแม็บเป็น Type1-Type4
   (fallback: "ประเภท" หรือ "Type")
5. "signed": true/false — ดูช่องลายเซ็นผู้กู้ยืมเงิน หากมีรอยหมึกแม้แต่จุดเดียวถือว่า signed=true

STRICT RULES:
- ตอบเป็น JSON object เดียวเท่านั้น ห้าม markdown/code fence
- ต้องมี key ครบ 5 ตัว: name, id_card_top, id_card_body, loan_type, signed
- id_card_top และ id_card_body ต้องอ่านแยกกันอิสระ ห้ามก็อปข้ามช่อง
- ถ้าอ่านไม่ได้ ให้ใส่ null ห้ามเดา
"""

ทำไมต้องแยก id_card_top / id_card_body?

เพื่อสร้างกลไก Cross-Validation — ถ้า AI สับสนหรือ hallucinate แล้วก็อปเลขบัตรจากช่องเดียวไปทั้งสองช่อง ระบบจะตรวจไม่พบความผิดพลาด ดังนั้น Prompt จึงเขียนย้ำ (บรรทัด 109-110) ว่าห้ามคัดลอกข้ามกัน เพื่อให้การเปรียบเทียบใน validate() มีความหมายจริง ๆ

ทำไม signed ต้องระบุว่า "lenient"?

โมเดล Vision LLM มักจะประเมิน "ลายเซ็นที่สมบูรณ์" อย่างเข้มงวดเกินไปและตอบ false แม้มีรอยปากกาอยู่ Prompt จึงระบุชัดเจนว่าจุดเดียวหรือรอยขีดเล็ก ๆ ก็นับเป็น signed=true เพื่อลด False Negative ที่ทำให้เอกสารที่เซ็นแล้วไปตกอยู่ในโฟลเดอร์ Manual โดยไม่จำเป็น

📂 4. การค้นหาไฟล์ (Folder/File Discovery)

find_newdocs_folders(source_dir: str) -> list

ใช้ glob.glob(source_dir + "/NewDocs*") เพื่อหาทุกโฟลเดอร์ที่ชื่อขึ้นต้นด้วย NewDocs (รองรับ NewDocs, NewDocs 2, NewDocs copy ฯลฯ) มี fast-path พิเศษ: ถ้าโฟลเดอร์ที่ผู้ใช้เลือกเอง (basename) ขึ้นต้นด้วย NewDocs อยู่แล้ว จะ return [source_dir] ทันทีโดยไม่ค้นลึกลงไป — ป้องกันกรณีผู้ใช้เลือกโฟลเดอร์ NewDocs ตรง ๆ แทนที่จะเป็นโฟลเดอร์แม่

find_pdfs(folder: str) -> list

แม้ชื่อฟังก์ชันจะเป็น find_pdfs (ชื่อเดิมจากตอนที่รองรับแค่ PDF) แต่ปัจจุบันมันเดิน glob ทุกไฟล์ในโฟลเดอร์ (glob.glob(folder + "/*")) แล้วกรองด้วยเงื่อนไข:

ext == ".pdf" or ext in IMAGE_EXTENSIONS

ผลลัพธ์ผ่าน sorted(set(...)) เพื่อลบไฟล์ซ้ำ (กรณี glob pattern คาบเกี่ยว) และให้ลำดับที่คงที่ (deterministic) ทุกครั้งที่รัน

🖼️ 5. กระบวนการย่อรูปภาพ (prepare_image)

prepare_image(image_path: str, max_dim: int = MAX_IMAGE_DIMENSION) -> bytes | None

ขั้นตอนการทำงาน:

  1. เปิดไฟล์ด้วย PIL.Image.open() แล้วแปลงเป็น RGB (กัน error กรณีไฟล์เป็น CMYK, Palette, RGBA ที่ Gemini อาจไม่รองรับ)
  2. คำนวณ longest = max(w, h)
  3. ถ้า longest > max_dim — คำนวณ scale factor แล้ว resize ด้วย Image.LANCZOS (อัลกอริทึม resampling คุณภาพสูงสุดใน Pillow เหมาะกับข้อความ/ลายเส้น)
  4. เขียนผลลัพธ์ลง io.BytesIO() เป็น JPEG คุณภาพ JPEG_QUALITY=85 พร้อม optimize=True (ลด byte เพิ่มโดยไม่เสียคุณภาพ)
  5. คืนค่าเป็น bytes ดิบ พร้อมส่งเข้า Gemini โดยตรง (ไม่ต้องเขียนไฟล์ชั่วคราวลงดิสก์)

Aspect ratio จะถูกรักษาไว้เสมอ (คำนวณ new_size จาก scale เดียวกันทั้ง w และ h) — ไม่มีการ crop หรือบิดเบือนภาพ

⚠️ ถ้าไฟล์เสียหายหรือ Pillow เปิดไม่ได้ ฟังก์ชันจะ catch exception, log ข้อความ [ERROR] Cannot read image ... แล้ว return None — ทำให้ extract_data รู้ทันทีว่าไฟล์นี้ไปต่อไม่ได้

🧠 6. ตรรกะการเรียก AI (extract_data)

extract_data(pdf_path: str) -> dict | None

ฟังก์ชันนี้เป็นจุดตัดสินใจแรกว่าไฟล์เป็น PDF หรือรูปภาพ โดยเช็คจาก os.path.splitext(pdf_path)[1].lower():

เส้นทางที่ 1 — เป็นรูปภาพ (ext in IMAGE_EXTENSIONS)

เข้าสู่ลูป for max_dim in (MAX_IMAGE_DIMENSION, MAX_IMAGE_DIMENSION_RETRY) — รายละเอียดในหัวข้อถัดไป (Auto-Retry)

เส้นทางที่ 2 — เป็น PDF

อ่านไฟล์ทั้งก้อนด้วย open(path, "rb").read() ตรง ๆ (ไม่มีการย่อขนาด เพราะ Gemini รองรับ PDF หลายหน้าโดยตรงและ PDF มักมีขนาดไฟล์เล็กกว่ารูปถ่ายอยู่แล้ว) แล้วส่งต่อด้วย mime type application/pdf

🔁 7. กลไก Auto-Retry ความละเอียด

นี่คือกลไกสำคัญที่แก้ปัญหาภาพถ่ายความละเอียดสูงจากมือถือ (เช่น 4536×8064) ที่เมื่อย่อครั้งเดียวแล้วตัวเลขอาจเบลอเกินอ่าน:

main.py:191-202
data = None
for max_dim in (MAX_IMAGE_DIMENSION, MAX_IMAGE_DIMENSION_RETRY):  # 2048, 3072
    file_bytes = self.prepare_image(pdf_path, max_dim)
    if not file_bytes:
        return None
    data = self._call_ai(file_bytes, "image/jpeg")
    if data is not None and self._ids_readable(data):
        return data          # สำเร็จ → คืนค่าทันที ไม่ต้อง retry
    if max_dim != MAX_IMAGE_DIMENSION_RETRY:
        self.log("  [RETRY] ID numbers unreadable at low resolution — retrying...")
return data                  # หมดรอบแล้วยังอ่านไม่ได้ → คืนผลลัพธ์รอบสุดท้าย (อาจมี field อื่นครบ)

เกณฑ์การตัดสินว่า "ต้อง Retry" — _ids_readable()

_ids_readable(data: dict) -> bool

ตรวจว่า ทั้ง id_card_top และ id_card_body เป็นตัวเลข regex \d{'{'}13{'}'} ครบทั้งคู่หรือไม่ ถ้าไม่ครบแม้แต่ช่องเดียว ถือว่า "อ่านไม่ได้" และไป retry ที่ความละเอียดสูงขึ้น

💡 การออกแบบนี้ช่วยประหยัด Token โดยเฉลี่ย เพราะเอกสารส่วนใหญ่อ่านผ่านตั้งแต่รอบแรกที่ 2048px (ราคาถูกกว่า) และจะ "จ่ายแพงขึ้น" เฉพาะไฟล์ที่จำเป็นจริง ๆ เท่านั้น (retry ที่ 3072px)

🚦 8. การจัดการ Rate Limit / Error (_call_ai)

_call_ai(file_bytes: bytes, mime_type: str) -> dict | None

ฟังก์ชันนี้คือจุดเดียวที่เรียก self.model.generate_content(...) จริง ๆ ทั้งสำหรับ PDF และรูปภาพ (ถูก refactor ออกมาให้ extract_data เรียกใช้ร่วมกัน) มีลอจิกดังนี้:

  1. วนลูปสูงสุด max_retries = 3 ครั้ง
  2. เรียก Gemini แล้วดึง response.text
  3. ใช้ regex re.sub(r"^```(?:json)?\s*|\s*```$", "", raw) ลบ markdown code fence ที่ AI อาจใส่มาโดยไม่ตั้งใจ (แม้ตั้ง response_mime_type="application/json" ไว้แล้วก็ตาม เป็นการป้องกันสองชั้น)
  4. json.loads(raw) แปลงเป็น dict แล้ว return

การตรวจจับ Rate Limit (HTTP 429)

ถ้า exception message มีคำว่า "429", "quota", หรือ "rate" (case-insensitive) ระบบจะ:

  • พยายามดึงเวลาที่ต้องรอจากข้อความ error ด้วย regex retry in ([\d.]+)s
  • ถ้าดึงได้ → รอ wait_time = ตัวเลขที่ดึงได้ + 2 วินาที (buffer กันคลาดเคลื่อน)
  • ถ้าดึงไม่ได้ → รอ default 40 วินาที
  • ใช้ time.sleep(wait_time) แล้ว continue ไป attempt ถัดไป
  • ถ้าครบ 3 ครั้งแล้วยังไม่สำเร็จ → log error และ return None

Error อื่น ๆ ที่ไม่ใช่ Rate Limit

จะถูก raise ต่อออกไปให้ block ข้างนอกจับด้วย except json.JSONDecodeError (JSON เพี้ยน) หรือ except Exception (error ทั่วไป เช่น network, API key ผิด) แล้ว log พร้อม return None

✅ 9. ตรรกะการตรวจสอบ (validate)

validate(data: dict) -> str | None

ฟังก์ชัน static ที่รับ dict จาก AI แล้วตรวจตามลำดับ (fail-fast — เจอผิดจุดแรกก็ return ทันที):

ลำดับเงื่อนไขข้อความ error ที่คืน
1data ไม่ใช่ dict"AI response is not a JSON object."
2name ว่างเปล่า/ไม่มี"Missing name."
3id_card_top ไม่ตรง regex 13 หลัก"Invalid top ID card: ..."
4id_card_body ไม่ตรง regex 13 หลัก"Invalid body ID card: ..."
5id_card_top != id_card_body"ID mismatch: top ... != body ..."
6loan_type ไม่อยู่ใน LOAN_TYPES"Unknown loan type: ..."
7signed is not True"Document is not signed."
Backward compatibility: โค้ดใช้ data.get("id_card_top") or data.get("id_card") — รองรับทั้งฟิลด์ใหม่ id_card_top และฟิลด์เก่า id_card (จาก Prompt เวอร์ชันก่อนหน้าที่มีฟิลด์เดียว) ป้องกัน crash หากมี response ค้างจาก cache หรือโมเดลรุ่นเก่า
🔒 หมายเหตุด้านความเข้มงวด: เงื่อนไข signed is not True ใช้ is not True แทน not signed โดยตั้งใจ — เพื่อให้ None (อ่านไม่ได้) และ False (ไม่เซ็น) ถูกปฏิเสธเหมือนกัน มีแค่ True ที่ชัดเจนเท่านั้นที่ผ่าน

📁 10. การเปลี่ยนชื่อและย้ายไฟล์

sanitize_filename(text: str) -> str

ลบอักขระที่ห้ามใช้ในชื่อไฟล์ Windows ด้วย regex [<>:"/\\|?*\x00-\x1f] แล้วยุบช่องว่างซ้ำด้วย re.sub(r"\s+", " ", text) — ออกแบบมาให้ไฟล์ที่สร้างบน macOS/Linux ยังเปิดได้ปกติเมื่อย้ายไป Windows

unique_path(path: str) -> str

ถ้าปลายทางมีไฟล์ชื่อซ้ำอยู่แล้ว จะเติม (1), (2), ... ต่อท้ายชื่อไฟล์ (ก่อนนามสกุล) วนจนกว่าจะเจอชื่อที่ไม่ซ้ำ — ป้องกันการเขียนทับไฟล์เดิมโดยไม่ตั้งใจ

move_to_manual(pdf_path, output_dir, reason: str)

สร้างโฟลเดอร์ {'{'}output_dir{'}'}/Manual (ถ้ายังไม่มี) แล้วย้ายไฟล์เข้าไปโดยคงชื่อไฟล์เดิม (ไม่เปลี่ยนชื่อ เพราะข้อมูลอาจไม่ครบ/ไม่ถูกต้อง) พร้อม log บรรทัด [MANUAL] ... -> Manual (เหตุผล)

move_to_loan_folder(pdf_path, output_dir, data: dict)

สร้างชื่อไฟล์ใหม่ตามรูปแบบ:

🧵 13. โมเดล Threading และ Queue

Tkinter ไม่ thread-safe — ห้ามอัปเดต widget จาก thread อื่นโดยตรง ระบบจึงใช้สถาปัตยกรรม Producer/Consumer:

Worker Thread (_run_worker) Main/GUI Thread ────────────────────────── ──────────────── processor.run(src, out) after(100, _poll_log_queue) ← เรียกซ้ำทุก 100ms └─ self.log(msg) │ └─ log_queue.put(msg) ───────────────► log_queue.get_nowait() │ finally: ├─ ถ้า msg == "__WORKER_DONE__" log_queue.put("__WORKER_DONE__") │ → เปิดปุ่ม Start, ปิดปุ่ม Stop ├─ ถ้ามี "[OK]" → self._stat_ok += 1 ├─ ถ้ามี "[MANUAL]" → self._stat_manual += 1 └─ เขียนข้อความลง log_box (CTkTextbox)
  • self.worker = threading.Thread(target=self._run_worker, args=(src, out), daemon=True) — ตั้งเป็น daemon thread เพื่อไม่บล็อกการปิดโปรแกรม
  • self.processor.stop_flag = threading.Event() — ปุ่ม Stop เรียก stop_flag.set() แล้ว loop ใน run() เช็คสถานะนี้ทุกรอบไฟล์/โฟลเดอร์เพื่อหยุดแบบนุ่มนวล (finish ไฟล์ปัจจุบันก่อน)
  • Progress bar คำนวณจาก (stat_ok + stat_manual) / stat_total ที่ประเมินไว้ล่วงหน้าตอนกด Start
